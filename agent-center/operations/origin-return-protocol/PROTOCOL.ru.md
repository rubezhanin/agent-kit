# Первое правило для агента: вернуть результат туда, где его попросили

Если вы только начали собирать себе агента, не начинайте с большой архитектуры.

Не с команды из десяти ролей.
Не с памяти.
Не с доски задач.
Не с красивой схемы.

Сначала проверьте одну скучную вещь: агент умеет возвращать результат туда, где его попросили?

Потому что в реальной работе всё часто ломается именно здесь.

Агент вроде что-то сделал. Файл где-то появился. Другой агент что-то проверил. Фоновая задача закончилась. Но человек, который попросил, сидит и не понимает:

- где результат;
- кто сейчас отвечает;
- что проверено;
- почему агент молчит;
- можно ли это уже использовать.

Для человека такая задача не выполнена. Она потеряна.

Файл про самый первый слой агентской операционки: **Origin Return Protocol**.

Если проще: у каждой задачи должен быть обратный адрес.

---

## Зачем это вообще нужно

Один агент ещё можно держать в голове.

Вы написали ему в Telegram, он ответил там же. Всё понятно.

Но добавьте хотя бы один дополнительный слой - и начинается путаница:

- агент создаёт файл;
- агент запускает долгую задачу;
- агент передаёт работу другому агенту;
- агент работает через cron;
- агент складывает задачу в Kanban;
- агент делает черновик, а другой его проверяет;
- агент пишет `готово`, но ничего не прикладывает.

И вот тут становится видно: одного умного ответа мало.

Нужен маленький рабочий порядок.

Не тяжёлая методология. Не корпоративный регламент. Просто правило, которое не даёт работе исчезнуть.

---

## Главное правило

Задача закрыта только тогда, когда результат вернулся в исходную точку запроса.

Не когда модель закончила отвечать.
Не когда файл создан.
Не когда карточка стала `done`.
Не когда другой агент написал что-то у себя.

А когда человек получил понятный итог:

```text
что сделано
где лежит результат
что проверено
что осталось заблокировано
```

Если этого нет, задача ещё не закрыта.

---

## Пять вещей, которые должны быть у задачи

Для каждой задачи агент должен знать пять вещей.

| Поле | Простой вопрос |
|---|---|
| `origin` | Откуда пришла задача? |
| `owner` | Кто отвечает за итог? |
| `artifact` | Где результат? |
| `status` | Что сейчас с задачей? |
| `return_path` | Куда вернуть финальный ответ? |

Вот и весь первый слой.

Пока этих пяти вещей нет, агент может делать работу, но система уже слепая.

---

## Мини-шаблон

Скопируйте это в инструкцию своего агента.

```yaml
origin:
owner:
request:
artifact:
status:
return_path:
verification:
final_message:
```

Пример:

```yaml
origin: Telegram DM
owner: main-agent
request: подготовить короткий MD-файл про первый слой работы агентов
artifact: ./ORIGIN_RETURN_PROTOCOL.md
status: DONE
return_path: исходный Telegram-чат
verification: файл создан, текст проверен, приватных данных нет
final_message: файл готов, приложен ниже
```

Смысл простой: если агент не может заполнить блок, он сам не понимает, где находится задача.

---

## Статусы без зоопарка

Не надо двадцать статусов. Хватит четырёх.

| Статус | Когда использовать |
|---|---|
| `DONE` | Результат готов, проверен и возвращён туда, где его попросили |
| `BLOCKED` | Агент не может продолжить без файла, доступа, решения или контекста |
| `NEEDS_APPROVAL` | Нужно действие с внешним эффектом: отправить, опубликовать, удалить, оплатить, изменить живую систему |
| `STALE` | Задача зависла или устарела, её надо перепроверить |

Самая частая ошибка - ставить `DONE`, когда агент просто закончил у себя.

Такой статус нельзя считать `DONE`. Это просто след где-то в системе.

---

## Что должен вернуть агент в конце

Хороший финальный ответ выглядит так:

```text
Status: DONE
Outcome: что сделано
Artifact: где лежит результат
Verification: что проверено
Returned to: куда вернул итог
Blocked: что осталось нерешённым, если есть
```

Плохой ответ:

```text
Готово.
```

Почему плохой: непонятно, что готово, где оно лежит и можно ли этому верить.

---

## Как это выглядит в одном агенте

Вы просите агента:

```text
Собери короткий отчёт по файлу и верни мне итог.
```

Агент должен держать контур:

```text
origin: текущий чат
owner: текущий агент
artifact: report.md или текст в ответе
status: DONE / BLOCKED
return_path: текущий чат
verification: файл прочитан, вывод сверён с запросом
```

Если он создал `report.md`, но не прислал путь и короткий итог, задача не закрыта.

---

## Как это выглядит в команде агентов

Например, у вас есть главный агент и агент-исследователь.

Плохо:

```text
Главный агент отправил задачу исследователю и забыл.
Исследователь что-то сделал в своём чате.
Пользователь ничего не получил.
```

Нормально:

```yaml
origin: Telegram DM пользователя
owner: main-agent
assignee: research-agent
expected_output: короткий файл с фактами и ссылками
artifact: ./research-notes.md
verification: ссылки открываются, выводы не выдуманы
return_path: исходный Telegram DM
status: DONE
```

Главный агент может не делать исследование сам. Но он должен вернуть итог человеку.

Иначе команды нет. Есть несколько окон, где что-то происходит само по себе.

---

## Когда агент обязан остановиться

Агент не должен молча делать действия с внешним эффектом.

Он должен вернуть `NEEDS_APPROVAL`, если надо:

- отправить сообщение от имени человека;
- опубликовать пост;
- удалить или перезаписать пользовательские данные;
- поменять живую настройку сервиса;
- потратить деньги;
- использовать секреты, токены, cookies или OAuth;
- написать клиенту, партнёру или внешнему человеку;
- принять юридическое, финансовое или медицинское решение без человека.

Это тоже часть обратного адреса.

Человек должен понимать не только где результат, но и где агент остановился.

## Как поставить это в Hermes как правило

Самый простой вариант - сохранить файл рядом с агентом и добавить в его инструкцию такой блок:

```markdown
## Origin Return Protocol

Для любой задачи, которая создаёт файл, запускает долгую работу, уходит в cron, Kanban, background или другому агенту, сохраняй:

- origin: откуда пришёл запрос;
- owner: кто отвечает за итог;
- artifact: где результат;
- status: DONE / BLOCKED / NEEDS_APPROVAL / STALE;
- return_path: куда вернуть финальный ответ.

Не закрывай задачу как DONE, пока результат не вернулся в return_path.

Финальный ответ всегда содержит:

Status:
Outcome:
Artifact:
Verification:
Returned to:
Blocked:
```

Если хотите именно как Hermes skill, положите файл в профиль агента:

```bash
mkdir -p ~/.hermes/profiles/<profile>/skills/operations/origin-return-protocol
cp ORIGIN_RETURN_PROTOCOL.md ~/.hermes/profiles/<profile>/skills/operations/origin-return-protocol/SKILL.md
```

Потом проверьте:

```bash
hermes --profile <profile> skills list --source local
hermes --profile <profile> chat -s origin-return-protocol -q "Коротко объясни, что такое return_path"
```

Если команда отличается в вашей версии Hermes, откройте список skills и проверьте путь профиля:

```bash
hermes --profile <profile> config path
hermes --profile <profile> skills list
```

## Быстрый тест

Дайте агенту задачу:

```text
Создай файл test-result.md с одной строкой: ORIGIN_RETURN_OK.
Используй Origin Return Protocol.
Верни мне статус, путь к файлу, проверку и финальное сообщение.
```

Хороший ответ:

```text
Status: DONE
Artifact: ./test-result.md
Verification: файл создан, строка ORIGIN_RETURN_OK проверена
Returned to: текущий чат
Blocked: нет
```

Плохой ответ:

```text
Сделал.
```

Если агент отвечает вторым способом, ему рано давать длинные цепочки, других агентов и фоновые задачи. Сначала надо научить его закрывать контур.

---

## Чеклист для человека

Когда агент пишет `готово`, спросите четыре вещи:

```text
Где результат?
Что проверено?
Куда он вернулся?
Что осталось заблокировано?
```

Если ответа нет, задача ещё не закрыта.

---

## Почему это первый слой

Память нужна.
Доска задач нужна.
Роли агентов нужны.
Проверки нужны.

Но если результат не возвращается в исходную точку, всё это начинает плодить шум.

Становится больше мест, где работа может потеряться.

Поэтому первый слой такой простой:

```text
запрос -> владелец -> работа -> результат -> проверка -> возврат
```

Сначала добейтесь этого.

Потом уже добавляйте доску задач, память, роли, проверки, handoff и нормальную команду агентов.

---

## Короткая формула

```text
Нет origin - агент не знает, откуда пришла задача.
Нет owner - никто не отвечает за итог.
Нет artifact - результат нельзя найти.
Нет verification - результату нельзя доверять.
Нет return_path - задача не завершена.
```

Вот с этого и стоит начинать агентскую операционку.
