# Управление контекстом AI-агента: полное руководство

> Как сделать так, чтобы агент не тупел, не забывал и работал стабильно в длинных сессиях.
> Основано на реальной эксплуатации системы из 7 агентов и 38 кронов (OpenClaw, Opus 4.6).

---

## Часть 1: Архитектура для человека

### Проблема

У каждой LLM есть контекстное окно - максимальный объём текста, который модель "видит" за раз. Для Claude Opus 4.6 это 200K токенов. Звучит много, но вот что его съедает:

```
System prompt (.md файлы)     ~35-50K токенов (грузится в КАЖДОЕ сообщение)
История диалога                ~2-5K за каждый обмен репликами
Tool calls (exec, read, web)   ~1-15K за каждый вызов
Tool results                   ~1-50K (особенно read файлов и web_search)
```

При активной работе контекст заполняется за 20-40 сообщений. После этого:
- **60-70%** - агент начинает "тупить", терять детали
- **~80%** - автоматическая компакция (принудительное сжатие) - потеря точности
- **90%+** - критическая зона, ответы деградируют сильно

### Решение: три уровня защиты

```
┌─────────────────────────────────────────────────┐
│           УРОВЕНЬ 1: ПРЕДОТВРАЩЕНИЕ             │
│                                                 │
│  Не тащить в контекст то, что не нужно          │
│  • Лёгкий system prompt (минимум .md в корне)   │
│  • read с offset/limit                          │
│  • exec с | head -50                            │
│  • Субагенты для тяжёлых задач                  │
│  • Запрет на полные дампы (config, cron list)    │
└─────────────────────┬───────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────┐
│           УРОВЕНЬ 2: МОНИТОРИНГ                 │
│                                                 │
│  Знать сколько контекста занято                  │
│  • session_status каждые 5-7 сообщений          │
│  • Автоматический context check (крон 30 мин)   │
│  • Пороги: 60% → 70% → 75% → 80%               │
└─────────────────────┬───────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────┐
│           УРОВЕНЬ 3: GRACEFUL SAVE              │
│                                                 │
│  Если контекст большой - сохраниться            │
│  • Handoff файл (save game)                     │
│  • Дневник сессии                               │
│  • /new с чистого листа (но с памятью)          │
│  • Bootstrap при пробуждении                    │
└─────────────────────────────────────────────────┘
```

### Уровень 1: Предотвращение (где больше всего экономия)

**System prompt = самый дорогой ресурс.** Каждый .md файл в рабочей директории грузится в КАЖДОЕ сообщение. Если у тебя 50KB в system prompt - это 50KB × каждый ход.

Что мы сделали:
- Вынесли детальные правила из AGENTS.md в `references/` (ссылки, не полный текст)
- AGENTS.md: 13.4KB → 8.5KB
- TOOLS.md: 7.3KB → 4.3KB
- HEARTBEAT.md: 5.7KB → 0.2KB
- **Итого: -13.4KB (-30%) из system prompt**

Принцип: **в корне только указатели, детали в references/**

**Субагенты = разгрузка основной сессии.** Тяжёлая работа запускается в отдельной сессии со своим контекстом. Основная сессия получает только результат.

Когда делегировать субагенту:
- Чтение файлов >100 строк
- Ресёрч с >3 запросами поиска
- Любой анализ с 3+ tool calls
- Генерация контента (посты, документы)

**Осторожное чтение.** Не `read file.md` целиком, а `read file.md offset=45 limit=20` - только нужные строки. Не `exec cat log.txt`, а `exec cat log.txt | head -50`.

### Уровень 2: Мониторинг

Агент сам проверяет `session_status` каждые 5-7 ответов и видит:
```
📚 Context: 74K/200K (37%) · Compactions: 0
```

Таблица реакций:

| Контекст | Что делать |
|----------|-----------|
| <60% | Всё ок, работаем нормально |
| 60-70% | Всё тяжёлое → субагентам, самому только координация |
| 70-75% | СТОП. Записать handoff + дневник. Предупредить пользователя |
| 75-80% | Сообщить: "⚡ Контекст 75%, рекомендую /new" |
| >80% | СТОП. Handoff → предложить /new. Не продолжать |

Дополнительно: крон каждые 30 минут автоматически проверяет контекст и предупреждает при >70%.

### Уровень 3: Graceful Save (Handoff)

Handoff - это "save game" перед сбросом сессии. Файл `memory/handoff.md`:

```markdown
# Handoff | YYYY-MM-DD HH:MM

## Тема
Что обсуждали / над чем работали

## Решения за сессию
- Решение 1
- Решение 2

## TODO (незавершённое)
- [ ] Задача 1
- [ ] Задача 2

## Файлы изменённые за сессию
- path/to/file.md

## Контекст для продолжения
Что нужно знать чтобы продолжить работу
```

После /new или перезапуска агент:
1. Читает `memory/handoff.md` (save game)
2. Читает `memory/YYYY-MM-DD.md` (дневник дня)
3. Сверяет с последними сообщениями (верификация)
4. Продолжает работу

**Разница handoff vs автоматическая компакция:**

| | Handoff (/new) | Компакция (авто) |
|---|---|---|
| Кто решает что сохранить | Агент, осознанно | Система, алгоритмически |
| Потери | Минимальные | Непредсказуемые |
| Качество после | 9/10 | 6/10 |
| Контроль | Полный | Нулевой |

### Что запрещено делать в основной сессии

- `config.get` без конкретного `path` - весь конфиг = ~50K токенов
- `cron list` без фильтра - список всех кронов = ~30K токенов
- Чтение changelog целиком - ~15-30K токенов
- Чтение больших файлов целиком (>100 строк)
- Более 3 вызовов web_search подряд
- Генерация длинного контента (посты, документы) - только через субагент

### Результаты

До оптимизации:
- System prompt: ~48KB
- Контекст 70% после 15-20 сообщений
- 2-3 /new за рабочий день
- Частые компакции с потерей контекста

После оптимизации:
- System prompt: ~33KB (-30%)
- Контекст 37% после стратсессии (40+ минут разговора)
- 1 /new за день или реже
- Компакции практически не случаются

## Часть 2: Инструкция для агента (copy-paste в AGENTS.md)

Скопируй блок ниже в свой AGENTS.md или аналогичный файл настроек агента.

---

```markdown
## Context Management

### Мониторинг
Каждые 5-7 ответов → `session_status` → проверка %.

| Контекст | Действие |
|----------|----------|
| >60% | Всё тяжёлое → субагентам |
| >70% | СТОП. Запиши handoff + дневник. Потом продолжай |
| >75% | Сообщи пользователю: "⚡ Контекст 75%, рекомендую /new" |
| >80% | СТОП. Handoff → предложи /new |

### Запрещено в основной сессии
- `config.get` без конкретного `path`
- Ресёрч >3 web_search подряд → субагент
- Чтение файлов >100 строк → субагент или read с offset/limit
- Любой анализ с 3+ tool calls → субагент
- Генерация контента (посты, документы) → субагент

### Минимизация контекста
- `read` ВСЕГДА с `offset/limit` — НЕ читать файл целиком ради одной секции
- `exec`: добавлять `| head -50` или `| tail -30` к командам с большим выводом
- Не перечитывать файлы которые уже в system prompt
- Результат субагента = краткая выжимка, не полный вывод

### System prompt: правило корня
Каждый .md файл в корне workspace = грузится в КАЖДОЕ сообщение.
В корне → только указатели (краткие файлы <3KB).
Детали → в `references/`, агент читает по необходимости.

### Handoff (сохранение перед сбросом)
При >70% контекста или перед /new → записать `memory/handoff.md`:

```
# Handoff | YYYY-MM-DD HH:MM

## Тема
[Что обсуждали]

## Решения за сессию
- [Решение 1]

## TODO (незавершённое)
- [ ] [Задача]

## Контекст для продолжения
[Что нужно знать чтобы продолжить]
```

### Bootstrap (восстановление после сброса)
После /new или компактификации → СРАЗУ:
1. `read memory/handoff.md` — save game
2. `read memory/YYYY-MM-DD.md` — дневник дня
3. Сверить с `sessions_history(limit=10)` — совпадает ли handoff с реальностью
4. Если handoff устарел → ориентироваться на sessions_history
5. Есть незавершённая задача → продолжить. Нет → молчать.

### Автоматический context check (крон)
Крон каждые 30 минут запускает проверку. Агент выполняет session_status:
- <70% → молчать (NO_REPLY)
- >70% → записать handoff, сообщить пользователю
- >80% → СТОП, handoff, предложить /new

### Компакция ≠ handoff
Автоматическая компакция сжимает контекст алгоритмически — теряет детали.
Handoff — осознанное сохранение агентом — минимальные потери.
**Всегда предпочитай handoff перед компакцией.**
```

## Часть 3: Настройка крона автопроверки контекста

Создай крон в OpenClaw (через агента или CLI):

```json
{
  "name": "⚡ Context Check",
  "schedule": {
    "kind": "every",
    "everyMs": 1800000
  },
  "sessionTarget": "main",
  "payload": {
    "kind": "systemEvent",
    "text": "⚡ CONTEXT CHECK (автоматический каждые 30 мин). Выполни session_status. Если >70% — запиши handoff + дневник и сообщи пользователю '⚡ Контекст N%, записал handoff, рекомендую /new'. Если >80% — СТОП, handoff, message пользователю '🔴 Контекст N%, нужен /new'. Если <70% — молчи (NO_REPLY)."
  }
}
```

## Часть 4: Чек-лист оптимизации system prompt

1. **Посчитай текущий размер:**
   ```bash
   wc -c *.md  # в корне workspace
   ```

2. **Всё что >3KB → вынеси в references/**
   - Оставь в корне только краткую версию со ссылкой
   - Пример: `Подробности → references/full-rules.md`

3. **Проверь что НЕ должно быть в корне:**
   - Changelog
   - Полные инструкции
   - Истории, примеры, шаблоны
   - Данные (таблицы, списки)

4. **Оставь в корне ТОЛЬКО:**
   - AGENTS.md (краткие правила + ссылки)
   - IDENTITY.md / SOUL.md (личность)
   - MEMORY.md (ключевые факты, <3KB)
   - TOOLS.md (краткий справочник инструментов)
   - BOOTSTRAP.md (инструкция пробуждения)
   - USER.md (профиль пользователя)

5. **Целевые размеры:**
   | Файл | Максимум |
   |------|---------|
   | AGENTS.md | 8-10KB |
   | IDENTITY.md | 3-4KB |
   | MEMORY.md | 3KB |
   | TOOLS.md | 4-5KB |
   | BOOTSTRAP.md | 1-2KB |
   | USER.md | 2-3KB |
   | **Итого system prompt** | **<35KB** |
